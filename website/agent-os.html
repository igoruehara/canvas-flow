Arquitetura agentica

Agent OS no Canvas Flow

O Canvas Flow usa um contrato interno para organizar papel do agente, regras, capacidades e delegacao. Isso resolve a orquestracao dentro do produto sem precisar de A2A para tarefas internas.

Modelo mental

O `agentSpec` e o manifesto do agente descrevem o que o agente e, o que ele pode usar e quais limites precisa respeitar.

agents.md

Define identidade, responsabilidades e arquitetura operacional.

Guardrails e rules

Separam limites duros de comportamentos obrigatorios.

Skills e subagents

Encapsulam especialidades e contexto isolado para delegacao controlada.

Delegacao interna

Quando uma tarefa envolve decisao, varias tools ou integracao externa, o Assistente IA pode materializar uma arquitetura com Agent Plan, agente orquestrador, skill especialista e node MCP.

Agent Plan

Registra a estrategia antes da execucao, incluindo dados de entrada, validacoes e criterio de sucesso.

Agente orquestrador

Escolhe a skill certa, respeita guardrails e decide quando chamar ferramentas.

Skill especialista

Recebe uma tarefa especifica, usa os MCPs permitidos e normaliza a resposta para o orquestrador.

Edges de manifesto

Representam dependencias agenticas sem transformar tudo em um caminho linear de execucao.

Quando A2A faria sentido

Para a arquitetura interna, o Agent OS ja cobre a funcao pratica de delegar para especialistas. A2A faria sentido como adapter de borda, quando um agente Canvas Flow precisar conversar com agentes independentes de outros runtimes ou quando agentes externos precisarem descobrir e chamar um agente Canvas Flow.

Resumo: Agent OS para orquestracao interna; MCP para tools; A2A apenas se houver interoperabilidade entre agentes externos.
